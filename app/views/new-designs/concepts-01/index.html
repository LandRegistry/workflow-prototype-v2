Workflow ‐ design concepts

Primarily looking at options to introduce a core information architecture (IA) structure, to easily accommodate growth of the service with additional services and to safely avoid compromised or convoluted user journeys.

As well as the information being grouped logically to aid usability, the IA should be effortlessly expandable (while also creating a structure to allow work on individual areas and pages/features interdependently).

Following is a series of concepts to support this via the user-interface. This IA is also combined with the opportunity to build the presentation layer soundly, so that it is extensible and futureproofed for GDS compliance.

Information architecture

User-interface devices to support changes in information architecture. Designs largely negate requirements for breadcrumbs.

Concept 1

  • View concept 1
Type
Left navigation
Description
Workflow content split into two conceptually in the menu. Nested secondary sub links are 'anchor' or 'jump-to' links, to land on the relevant section of the page. Simple use of top level 'My Worklist' menu link. (Can add soft scroll). Sub links disappear when not on the 'Worklist' page: adds additional 'click' to return to the Worklist page, but is simple to interpret.

Left navigation
Description
Workflow content split across two pages with nested secondary sub links targeting an 'applications' and 'progress later' pages. This creates a 'My Worklist' landing page - and an opportunity to create a default landing/home page for the 'Workflow' service (which can provide dashboard info). Not suitable for use as a secondary navigtion (with a top-level navigation).

'Applications' and 'Progress later' links disappear when not in the 'My Worklist' area.
Concept 2b
This alternative variation (2b) persists the secondary links when elsewhere in the service. Included here for completeness. 'Applications' and 'Progress later' links do not disappear when not in the 'My Worklist' area.

Top level and secondary navigation
Description
A primary navigation (in the page header) sets out the most fundemental information groupings: which have their own secondary navigation (left nav).

These top level options can either serve their own 'area' landing page or default to the first page in the secondary navigation (as in this example).

Potentially move 'Search' from top level nav and include in secondary nav, so that each area has its own set of search options.

'Report an issue' either moved to the footer (or left nav).

We can incorporate 'Workflow' banner title/heading above the navigation and/or a 'Home' link in the top level nav if a landing dashboard was ever required.

Worklist, is supported by two secondary links: 'applications' and 'progress later'.
