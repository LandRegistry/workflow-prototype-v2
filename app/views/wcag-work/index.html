WCAG / Accessibility templates

These HTML templates (with revised HTML) featuring the HTML to resolve recorded accessibility and GDS compliance issues. Grouped by page (areas).

These updates and amends are the bare minimal to comply with accessibility requirements and GDS compliance which is accessibility related. (Improvements in the HTML and design is beyond the scope of this work).

Manage assigned work - Results/Banners/Error message pages

All error message pages should feature:

  • Prepended text "Error:" to the HTML document title (title tag)
  • Error summary block with hyperlinks which point to the related form element
  • Inline error messages, with correct styling (red left bar and text input fields with red border) and the corresponding 'aria-describedby' attributes
  1. This page is Work In Progress ... No user found: In the 'Find' input field enter and submit a name which is not recognised - a pink banner displays stating "No Workflow user found with the name ... ." Instead:
    • results-no-user-found
  2. 'Find' is a required field: Submiting the 'find' option without a value generates a pink banner displaying 'No Workflow user found with the name "" '. This is a form validation message - when no value is entered - for a required field and instead should be presented as an validation error with this message, as per the design:
    • error-enter-a-value-for-find.
  3. Submiting 'Assign' - 'Find' is a required field: The result is similar to the previous 'error-enter-a-value-for-find' example - when the 'find' field is empty and a user enters a 'assign' value and submits the 'assign' field the error message for the empty 'find' field is also fired. This example shows the same error message with a value in the 'assign' field:
    • error-enter-a-value-for-find-submitting-assign.
  4. Submiting 'Assign' with no values in either fields: If the user submits the assign field without entering any value into both the 'find' and 'assign' field, this is how the error messages appear:
    • error-enter-a-value-for-find-and-assign.
  5. 'Assign' is a required field: The logic of how this page works requires redesigning but in order to get the current design to be accessibile (and GDS compliant) we need to use the correct error messaging patterns.

    If 'Find' has a value, then submiting the 'Assign' field without a value (the first time, before a list of applications has been returned) currently generates a pink banner displaying "Select an application and find a caseworker to assign the application to." This is a form error validation message (required fields) - this message and design is required:
    • error-enter-a-value-for-assign.
    The summary message hyperlink for selecting an application should target the first application checkbox (not the 'Select all' checkbox option). When the 'Select one or more applications to assign' error message is shown we also add the 'aria-describedby' attribute to each checkbox (as per the HTML design), except for the first 'Select all' checkbox option.
  6. Applications are required: When both 'find' and 'assign' have values and no application has been selected: this is how the page appears:
    • error-enter-a-value-for-applications.
  7. If form validation error messages have been fired for either 'assign' and/or 'applications' the user could theoretically manually clear the value in the 'find' field; so the first error message (error-enter-a-value-for-find) is fired. But if the user has successfully returned a list of applications and then clears all fields and 'clicks' the 'assign' button, this example provides an illustration of how all three error messages appear in this theoretical edge case:
    • error-all.
  8. Form expired banner: When the page/form has expired an 'important' banner (GDS notification banner component) message is served at the top of the page which states: "The form you were submitting has expired. Please try again.". The styling on this banner isn't quite correct - the HTML should be as per this:
    • banner-form-expired
