Flagging UI feature

The design: functional prototype (opens in new tab)

A new 'Add flag' "link" (button) is now added to the end of the action buttons for each case shown on the Worklist page (for both progress now and progress later lists): the "Add flag" button, lets caseworkers mark cases that need attention or follow-up. Clicking the button shows a flag icon next to that case and updates the button text (value) to "Remove flag", with the change saved immediately so it persists if the caseworker leaves and returns to the page. The flag state is also stored in a database, so it's consistent across devices. For users without JavaScript, the page reloads after clicking to apply the same update.

For mulit-titles, flags can only be added to 'parent/lead' case. (Flags can not be added to the attached applications).

How the flagging functionality works

Technical description

At the end of each row of actions (for each case) is a flag button (styled as a link) immediately followed by an SVG flag icon. The flag mechanism is a progressive enhancement layered over the standard table markup: the SVG is hidden by default (aria-hidden="true", with no hmlr-flag-icon-show class added) and the JavaScript toggles its visibility when clicking the button.

The JavaScript runs on DOMContentLoaded: on page load, it selects all .hmlr-flag-icon-btn elements within #main-content, then for each one reads any saved flag state from localStorage (keyed as flag_ plus the button's ID, with a 365-day expiry time and stored in JSON) and applies it to the DOM via applyFlagState() — showing or hiding the SVG, updating the title text for screen readers, and setting the button value/label to "Add flag" or "Remove flag".

When a user clicks a flag button, the handler calls e.preventDefault() to stop the form submitting (so the page never reloads), toggles the SVG visibility, ARIA state, title text, and button label, then immediately writes the new state to localStorage (within the browser). The final step — currently commented out in the code — calls updateFlagInDatabase(), which sends an asynchronous fetch POST to /api/update-flag with the case's record ID and flag state as JSON. The UI updates instantly without waiting for the server. If the server call fails, the .catch() block rolls everything back — reverting localStorage, the button label, the SVG visibility, and the ARIA attributes — so the ID never shows a flag state the database doesn't hold. The button is also disabled (pointerEvents = 'none') during the request to prevent double-clicks.

Basically, localStorage gives instant per-browser persistence. The AJAX call syncs to the server as the source of truth, and the error handler keeps the two consistent. When the page is reloaded from the server the updated HTML is served with the correct settings for each flag per case.

The 'flag' is only shown to the caseworker (and a leader if they log in to their workflow list); referring and deferring does not 'carry' the flag. The flag is set back to zero/no-flag: so if the case is returned, the flag does not 'remember' and previous states. (Set to default zero/no-show flag).

If it has been flagged, this is also visible in the manage assigned list. (No link, just the icon). [Design to follow].

'Flagging' is not recored in the audit history.

Non-JavaScript and WCAG fallback

Accessibility support

The JavaScript functionality is a progressive enhancement. In browsers which do not support this JavaScript, the click event on the button is never set. Therefore the button will simply submit the local form (wrapping the flag button) and the page reloads (having submitted the form) in the conventional way. (We update the database and reload the page with the HTML setting that flag appropriately: showing or hiding the SVG flag, updating the title text for screen readers, and setting the button value to "Add flag" or "Remove flag".)

Notes on implimentation

Some suggested dev notes

View the HTML page source to access the JavaScript and additional CSS (bottom of the page/file). Note the IDs progress_now_table and progress_later_table on the two HTML tables. (Each "link" - technically a button - is seperated by a hidden 'pipe'. Ensure this new button is also seperated from the previous link/button using a 'pipe'. View the HTML source).

To make the flag button update the remote database without a page reload - a suggestion:

  1. Ensure you have IDs progress_now_table and progress_later_table on the two HTML tables respectively (as per the prototype).
  2. Uncomment the database sync line in the click handler: //updateFlagInDatabase(recordId, isFlagged, this);
    (Remove the //).
  3. Create the server-side endpoint at POST /api/update-flag. It must:
    • Accept a JSON body of the shape { "recordId": "E365VWW", "isFlagged": true }
    • Update the flag column for that case record in the database
    • Return a 200 response with a JSON body (e.g. { "success": true }) — the client calls response.json(), so the response must be valid JSON
    • Return a non-2xx status on failure, which triggers the client-side rollback
  4. Probably best to add CSRF protection. The fetch request currently sends no CSRF token, but the page's forms use one. Either:
    • Add a <meta name="csrf-token" content="..."> tag to the page head, and
    • Uncomment the 'X-CSRF-Token' header line in updateFlagInDatabase(), then validate the token server-side
  5. Verify the record ID mapping. The browser/JavaScript gets the recordId by stripping add_flag_ from the button ID (giving the case reference, e.g. V667KXL). Ensure the database lookup uses the case reference, or change the markup/JavaScript if the database keys on a different identifier (e.g. the numeric row ID).
  6. Decide the source of truth on page load. Currently the initial flag state comes from localStorage, which is per-browser. But for cross-device consistency (if the user switches browsers or machines and/or a colleague logs into the worklist on their own PC), render the flag state server-side (add hmlr-flag-icon-show and set the button value in the template from the database). i.e, when the page first loads we check the database settings and serve the initial HTML page accordingly.
  7. Test the failure path. Temporarily make the endpoint return a 500 and confirm the UI reverts: SVG hides, button label restores, localStorage rolls back, and the button becomes clickable again.
  8. Q: Should we show errors to the user? The JavaScript block only logs to console. We could uncomment and show notifications to caseworkers so they know a flag failed to save? (If so, we should also mimic this with an important notification banner for Non-JavaScript users).
  9. Note: there are two options at the start of the JavaScript to temporarily un-comment to clear (reset) the localstorage of the flags if you need to, while developing. Un-comment one (the first line), reload the page once, then comment it back out. It only removes keys beginning with flag_, so anything else in localStorage (like its auto-stored session data) is left alone. If you prefer to nuke everything in localStorage for the page, use the second line.

Some UI ACs

Some acceptance criteria points
  1. Thew new 'Add flag' buttons are seperated from the previous link/button using a hidden 'pipe'. (As per the pre-existing links/buttons in the row)
  2. The database values set (override) the UI as the primary source of truth
  3. When the flag button is clicked, the following are updated:
    • SVG flag icon becomes visible and the aria-hidden attribute is updated to match
    • Title element within the SVG block is updated. (This tooltip/screen-reader text on the icon changes to reflect the new state)
    • Button value/label - toggles between "Add flag" and "Remove flag"
    • The cursor style is set to 'none' while the request is in flight, then restored afterwards
    • If the server request fails, all of the above (except the SVG and label changes that already happened) get reverted back to their previous values via the .catch() block
    • The support for localStorage works: when setting a flag to a different state (without a connection to the server or database), refreshing the page correctly shows the latest setting
    • Failure to update the database, rolls back any changes to the UI (if necessary)
  4. Keyboard tabbing works as expected
  5. Non-JavaScript functionality is supported
